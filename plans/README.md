## Findings

- Accepted ticker typos fragment the book. `normalize_instrument()` only uppercases the raw ticker, and `BookBuilder` groups by exact `instrument_id`, so `PETRO27`, `PETROO27`, `PETRRO27`, and similar accepted regex matches become separate books. The implementation uses a session-scoped valid ticker pool with explicit aliases and no fuzzy matching.
- The repo has no extraction quality metric harness. Current tests verify the happy-path fixture outputs but do not report ticker accuracy, false corrections, or rejection quality.
- Fast fuzzy/LLM extraction is a valid roadmap direction, but it should be gated behind deterministic canonicalization and metrics. V1 docs explicitly choose deterministic extraction; future adapters are allowed, but not as unmeasured runtime behavior.
- LLM provider configuration belongs in the backend, with the dashboard acting as a control plane only. Browser-owned API keys or arbitrary user-entered model server URLs would create security and correctness risk.
- The simulator is too clean for fuzzy extraction work; V1.2 needs controlled chaotic positives, hard negatives, and opt-in frontend chaos controls before enabling fuzzy matching.
- Bounded fuzzy matching can recover obvious session-pool near misses, but only after false-merge metrics exist.

## Implementation notes

- Plan 005 added chaotic positive, hard-negative, and rejection fixture categories plus simulator chaos controls.
- Plan 006 enabled bounded fuzzy matching against the session valid ticker pool only.
- `PETOR27` merges to `PETRO27` only after `PETRO27` exists in the session pool.
- `PETR27` and `BOVE26` are fuzzy exclusions; `BOVE26` must not merge into `BOVA26` before explicit acceptance.
- Expanded evaluator output reports `exact_row=27/27` and `false_merge=3/3` on the current fixture set.

## Findings considered and rejected

- General fuzzy matching directly in V1 extraction: rejected for now because it can merge real instruments incorrectly without an instrument universe and evaluation harness.
- Full instrument master in this iteration: rejected because V1 docs say not to add a full instrument master, and an alias map is enough to address the observed issue.
- Persistent valid ticker pool: rejected for V1 because the current backend is intentionally in-memory.
- LLM fallback as immediate runtime behavior: rejected until plan 002 establishes quality metrics and plan 003 defines latency, confidence, and fallback gates.
- Dashboard-side extraction: rejected because API keys, provider URLs, timeout policy, validation, and book updates must remain backend-owned.
- Unmeasured fuzzy matching: rejected because wrong ticker merges corrupt the book more severely than missed corrections.
